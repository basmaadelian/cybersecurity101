// 01

Course Methodology

Anchor to Real Industry Standards

This course is not built from intuition. Every topic, every session, and every hands-on task maps directly to what the industry's most respected entry-level certification bodies actually test and teach. The three primary references are OffSec PEN-200 (the course behind OSCP, the gold standard for practical penetration testing), INE's Penetration Testing Student path (PTSv2) leading to the eJPT v2 certification, and TCM Security's PJPT and PNPT paths. Every tool listed in this course appears in at least two of those three reference curricula. Tools were not added because they are interesting or new — they were added because a junior pentester will encounter them in job descriptions, in real engagements, and in the exams that get them hired.

Fixed Target Student Profile

The student profile is fixed: an absolute beginner with general IT literacy but no security background. This has three direct consequences. First, the course cannot open with exploitation — attacking without understanding how things work produces dangerous skill gaps. Second, the difficulty curve is deliberate and visible: each module assumes the previous one is complete. Third, the student needs to see themselves in the outcome, so the certification roadmap and capstone debrief exist to show that the destination is real and achievable.

Three-Pillar Sequence with a Simulation Block

The course is built on three sequential pillars followed by a real-world simulation block. Fundamentals come first because network pentesting builds directly on what is just learned: ports, protocols, services, operating systems, and credentials. Web pentesting is placed after network because it introduces a new attack surface, a new toolset centered on Burp Suite, and a different vulnerability taxonomy in the OWASP Top 10. Students arrive at web hacking with mature enumeration instincts and comfort with shells. The final simulation block removes the instructor from the equation entirely and asks the student to perform realistic end-to-end engagements independently, then communicate results professionally.

// 01.2

Engagement Lifecycle

The penetration testing lifecycle that this course teaches students to execute end-to-end:

01
Scoping
Define targets, rules of engagement, and legal authorization before any testing begins.
02
Reconnaissance
Passive and active information gathering. OSINT, DNS, WHOIS, and network mapping.
03
Enumeration
Deep service interrogation. Identify versions, users, shares, and potential entry points.
04
Vuln ID
Map discovered services to known CVEs, misconfigurations, and exploitable weaknesses.
05
Exploitation
Gain initial access. Execute payloads, abuse services, and establish a foothold.
06
Post-Exploit
Enumerate the compromised host. Hunt credentials, pivot paths, and sensitive data.
07
Priv Escalation
Move from low-privilege to root or SYSTEM. Exploit misconfigs, tokens, and weak services.
08
Reporting
Document every finding with evidence, impact, and actionable remediation recommendations.
// 02

Lab Targets

The three real-world lab engagements in Module 8 are not single-machine CTF challenges. Each is a network-level simulation involving multiple hosts, real attack paths, and a full professional engagement process from discovery to report. Students operate without guidance in all three.

// Lab 01 — Network
Linux Network
Environment
Architecture Multiple Linux hosts on an internal network segment. Not individual machines — a full network with inter-host relationships and layered access.
Entry Point Unknown. Student must perform full discovery and enumeration from a foothold machine with no prior knowledge of the network.
Phases Discovery → Enumeration → Vulnerability ID → Exploitation → Post-Exploitation → Privilege Escalation → Pivoting → Evidence Collection
Methodology This lab is delivered with explicit methodology narration — the student explains their thinking at each decision point, making the structured approach transferable.
Goal Root access on all reachable hosts. Full post-exploitation evidence collected and documented.
// Lab 02 — Active Directory
Active Directory
Network Environment
Architecture Domain Controller, member servers, and workstations. A realistic enterprise AD environment with domain trust relationships and service accounts.
Entry Point Low-privileged domain user credentials simulating a breached-environment scenario. Student starts with a foothold, not from outside.
Phases Network Discovery → AD Enumeration (BloodHound/PowerView) → Credential Capture → Lateral Movement → Privilege Escalation → Domain Admin Compromise
Key Skills LLMNR poisoning, NTLM relay, Kerberoasting, AS-REP Roasting, Pass-the-Hash, attack path visualization, DCSync awareness.
Goal Domain Admin compromise. Full kill chain documented with evidence at every step.
// Lab 03 — Web Application
Web Application
Assessment
Architecture A multi-vulnerability web application requiring vulnerability chaining to achieve meaningful impact. No single finding wins — the chain is the goal.
Entry Point Unauthenticated external access. Full black-box assessment from zero knowledge of the application.
Phases Reconnaissance → Technology Fingerprinting → Endpoint Discovery → Vulnerability Identification → Exploitation → Post-Exploitation → Evidence Collection
Key Skills Multiple OWASP Top 10 categories in one target. Students must identify, prioritize, and chain findings using Burp Suite, Gobuster, ffuf, sqlmap, and manual techniques.
Goal Remote code execution on the underlying server. Complete professional pentest report with CVSS-rated findings and remediation recommendations.

  • File upload vulnerabilities: why unrestricted file uploads are critical severity findings

  • SSRF: how applications make outbound requests and how attackers hijack those requests to reach internal services
  • Basic SSRF: probing internal network services and localhost ports through a vulnerable application
  • Blind SSRF: detecting SSRF when there is no direct response using out of band techniques
  • SSRF against cloud metadata endpoints: AWS IMDSv1 and retrieving instance credentials
  • OS Command Injection: how web applications call system commands and how unsanitized input reaches the OS

  • Inferential SQL injection: boolean based blind and time based blind techniques

CSRF
  • What CSRF is: forging authenticated requests on behalf of a victim user without their knowledge
  • How CSRF exploits the server's trust in the browser's automatically attached credentials
  • Conditions required for a CSRF attack to succeed
  • Crafting a CSRF proof of concept: HTML forms, image tags, and JavaScript based CSRF
  • CSRF with JSON requests and how content type restrictions affect exploitability
  • SameSite cookie attribute: Strict, Lax, and None and how each affects CSRF exploitability
  • CSRF tokens: common implementation flaws that make them bypassable
  • Chaining CSRF with XSS to bypass SameSite and token protections
Lab PortSwigger Web Security Academy — CSRF labs.
S-25
XSS — Cross Site Scripting Standalone
  • The browser trust model and why XSS is possible
  • Reflected XSS: unsanitized input immediately returned and executed in the victim's browser
  • Stored XSS: persistent payloads that execute for every user who visits the affected page
  • DOM based XSS: client side JavaScript sinks that execute attacker controlled input without server involvement
  • Real world XSS impact: session token theft, account takeover, keylogging, redirection, and defacement
  • Bypassing common XSS filters and WAF rules: encoding, polyglots, and context awareness
  • XSS in different HTML contexts: attribute injection, JavaScript injection, and URL injection
  • Chaining stored XSS with CSRF bypass
Lab PortSwigger Web Security Academy — XSS labs including DOM based and stored labs.
S-26
XXE — XML External Entity Injection Standalone
  • What XML is and how XML parsers work: entities, DTDs, and external entity resolution
  • Classic XXE to file read: retrieving sensitive server files through a vulnerable XML parser
  • XXE to SSRF: using external entity resolution to probe internal network services
  • Blind XXE: detecting and exploiting XXE when there is no direct output using out of band techniques
  • XXE via file upload: exploiting applications that parse XML in uploaded files such as SVG, XLSX, and DOCX
  • XXE in SOAP web services and API endpoints that accept XML
  • Mitigations: disabling external entity processing and why it is not always correctly implemented
Lab PortSwigger Web Security Academy — XXE labs including blind and file upload based labs.
S-27
Web Application Full Attack Chain Capstone Lab
  • Phase 1 — Reconnaissance: technology fingerprinting, directory enumeration, subdomain enumeration, and endpoint discovery
  • Phase 2 — Vulnerability identification: mapping discovered endpoints to vulnerability classes without firing a single exploit
  • Phase 3 — Exploitation: chaining multiple vulnerabilities to achieve meaningful impact
  • Phase 4 — Post exploitation: what to do after achieving code execution on the web server
  • Phase 5 — Evidence collection: documenting every step in real time as a professional engagement requires
  • Student works through a multi-vulnerability application entirely without guidance
Lab Full unsupported web application attack chain on a deliberately vulnerable application. Debrief covers the optimal attack chain, divergence points, and how all findings would be rated and written in a professional report.
